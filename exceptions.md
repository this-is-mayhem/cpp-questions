# Исключения в деструкторе
Деструкторы вызываются в ситуациях аварийного завершения (например, при разворачивании стека во время исключения). Если в деструкторе тоже будет выброшено исключение, а в это же время уже обрабатывается другое, произойдёт `std::terminate`.

Если нужно обработать ошибку, то поймать внутри деструктора и "поглотить" или залогировать:
```
struct Safe {
    ~Safe() noexcept {
        try {
            cleanup();
        } catch (const std::exception& e) {
            std::cerr << "Error in destructor: " << e.what() << "\n";
        }
    }
};
```

# `noexcept`
Это спецификатор, которым программист явно говорит компилятору и другим разработчикам: "Эта функция гарантированно не выбрасывает исключений."

Зачем нужен `noexcept`:
1. Оптимизация. Если компилятор знает, что функция точно не бросает исключений, он может:
   - генерировать более быстрый код (например, не вставлять защитные конструкции вокруг вызова);
   - использовать такие функции в местах, где исключения недопустимы (например, при оптимизациях контейнеров STL).

2. Семантика безопасности.
Например, операции перемещения (`move constructor`, `move assignment`) должны быть `noexcept`, чтобы стандартные контейнеры могли безопасно использовать их при расширении памяти.
Если они не `noexcept`, `std::vector` при `realloc` вместо перемещения будет копировать элементы, что медленнее.

    То есть:
    Когда `std::vector` нужно расширить память, то выделяется новый блок и для каждого старого элемента вызывается конструктор на новом месте:
    ```
    new (new_data + i) T(/* что-то */);
    ```
    Это "что-то" — либо **копия**, либо **перемещение** из старого элемента. Поэтому вопрос: как именно создавать новые? через `copy` или через `move`?
    
    STL здесь использует `std::move_if_noexcept`:
    ```
    for (size_t i = 0; i < old_size; ++i) {
    new (new_data + i) T(std::move_if_noexcept(old_data[i]));
    }
    ```
    Если у `T` есть `noexcept move constructor`, то вызывается `move`. Если `move` может бросить, вызывается `copy`.


3. Явное намерение.
`noexcept` — это обещание программиста: "Я продумал логику, тут исключений не будет". Это полезно и для читаемости кода.

## Как решить, ставить ли `noexcept`?

Обычно ставим `noexcept`, если:

- Функция простая: геттеры, сеттеры, операции над примитивами.
- Функция не делает ничего, что может бросить (выделение памяти, вызов чужого кода без гарантии `noexcept`, явный `throw`).
- Это move-конструктор или move-оператор (по современному "правилу пяти") — почти всегда `noexcept`.

Не ставим `noexcept`, если:
- В функции используется что-то, что потенциально может кинуть исключение (например, `new`, `std::string::append`, доступ к `std::map` и т.п.).
- Вы сами планируете бросать исключения для сигнализации об ошибке.

## Проверка `noexcept`
Можно спросить у компилятора, бросает ли что-то исключение:
```
static_assert(noexcept(std::declval<std::vector<int>>().clear())); // true
static_assert(!noexcept(std::declval<std::vector<int>>().push_back(1))); // false
```
Многие стандартные функции имеют аннотации `noexcept` в своей сигнатуре.
Пример: `std::vector::clear()` — это `noexcept`, а вот `push_back()` — нет.

Можно проверить так:
```
std::is_nothrow_invocable_v<decltype(&Foo::bar), Foo>
noexcept(expr) // ключевое слово, проверяет конкретное выражение
```

## Что обычно всегда помечают `noexcept`
1. **Деструкторы**. Стандарт делает деструкторы `noexcept` по умолчанию, кроме случаев, где есть исключения в базовых/членах.
    ```
    ~Foo() noexcept = default;
    ```

2. **Move-конструктор и move-оператор**. Иначе `std::vector` и другие контейнеры будут использовать копирование, а не перемещение → тормоза.
    ```
    Foo(Foo&&) noexcept;
    Foo& operator=(Foo&&) noexcept;
    ```

3. **Операции сравнения и простые inline-функции**.
    ```
    bool operator==(const Foo& other) const noexcept { return id == other.id; }
    int getId() const noexcept { return id; }
    ```

4. **Функции, где логика очевидна и нет аллокаций/исключений**. Например, работа с POD-типами, арифметика, побитовые операции, простая логика без new/string/map.


# Уровни exception safety
### 0. No exception safety
Если функция бросает исключение, объект может оказаться в неконсистентном состоянии (разрушены инварианты, утечки ресурсов и т.п.).
Так писать не принято.

### 1. Basic exception safety guarantee
При исключении ресурсы не теряются (нет утечек памяти, файлов и т.д.).
Объект остаётся в валидном, но неопределённом состоянии (инварианты сохранены, но данные могут измениться).
Программа может продолжить работу.

Пример:
`std::vector::push_back()` даёт базовую гарантию.
Если `push_back` упал на аллокации памяти, вектор остаётся валиден, но его содержимое может быть частично изменено.

### 2. Strong exception safety guarantee
При исключении операция откатывается так, будто её не вызывали. Объект остаётся в прежнем состоянии. Это похоже на транзакцию: “либо всё успешно, либо ничего”.

Пример:
`std::vector::emplace_back()` может давать сильную гарантию, если конструктор элемента не бросает исключений. Многие алгоритмы STL тоже стараются её предоставлять.

### 3. No-throw guarantee
Функция гарантированно не бросает. Обычно помечается как `noexcept`.

Примеры: Деструкторы (почти всегда noexcept). Move-конструкторы, если это возможно.
