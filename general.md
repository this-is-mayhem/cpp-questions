# `static`
## Внутри функции (локальная переменная)
```
void foo() {
    static int counter = 0; 
    counter++;
    std::cout << counter << "\n";
}
```
- Переменная создается один раз и сохраняет свое значение между вызовами функции. Локальная для функции `foo`. Место хранения: в статической памяти (не на стеке и не в куче). Обычно это сегмент .data (если инициализирована) или .bss (если неинициализирована).
- Инициализация происходит только при первом вызове.
- Живет до конца программы.
  
## Для глобальных/namespace переменных и функций
```
static int globalVar = 42;

static void helper() {
    // доступна только внутри этого .cpp файла
}
```
- Ограничивает область видимости текущим файлом (internal linkage).
- Обычно применяют, если не хотят "светить" переменную или функцию наружу.
  
## Внутри класса (статические члены класса)
```
struct MyClass {
    static int count; // общая переменная для всех объектов
    static void hello() { std::cout << "Hello\n"; }
};

int MyClass::count = 0; // нужно определить вне класса
```
- Общие для всех объектов (одна копия на весь класс).
- Можно вызывать без объекта:
  ```
  MyClass::hello();
  MyClass::count = 10;
  ```
Статические методы в классе:
- Не имеют неявного указателя `this`.
- Можно вызывать без объекта.
- Используются, например, как фабрики, счетчики, вспомогательные функции.

### Определение статического члена вне класса
Компилятор видит только объявление: "в классе будет такой-то статический член". Но никакой памяти под саму переменную ещё не выделяется. Это как «обещание», что где-то в программе будет одно единственное хранилище `int counter`, общее для всех объектов. Чтобы переменная действительно существовала в памяти, нужно её определить в одном `.cpp` файле. Без этого линковщик выдаст ошибку, потому что на этапе линковки он ищет реальный объект (память), а находит только объявления.

🔹 **Исключения (начиная с C++17)**

С появлением `inline static` можно писать прямо в классе:
```
struct MyClass {
    inline static int counter = 0;
};
```
Такое определение считается "inline" и может встречаться в нескольких единицах трансляции без конфликтов. Поэтому в modern C++ отдельное определение в `.cpp` уже часто не требуется.

## ❗ Анонимное пространство имен
В современных C++ вместо `static` для скрытия символов часто используют:
```
namespace {
    int hiddenVar = 42;

    void hiddenFunc() {
        std::cout << "Hi\n";
    }

    struct HiddenType {
        int x;
    };
}
```
Всё, что объявлено внутри него, видно только в текущем translation unit (т.е. внутри одного .cpp файла). Снаружи - доступ закрыт.

# `intN_t`, `uintN_t`
Беззнаковые:
- `uint8_t` — 8 бит (0–255)
- `uint16_t` — 16 бит (0–65535)
- `uint32_t` — 32 бита (0–4 294 967 295)
- `uint64_t` — 64 бита (0–18 446 744 073 709 551 615)

Знаковые:
- `int8_t` — 8 бит (−128…127)
- `int16_t`, `int32_t`, `int64_t` - аналоги беззнаковых с тем же числом бит.

Также есть типы с "как минимум" такой шириной:
`uint_least8_t`, `int_least16_t` и т.д. - минимальная ширина не меньше указанной.

`uint_fast8_t`, `int_fast32_t` - самые быстрые типы с не меньшей шириной.

Применение:
- `intN_t` / `uintN_t` - для бинарных форматов, сетевых протоколов, сериализации, микроконтроллеров.
- `_least` и `_fast` - для кроссплатформенных библиотек, где важна производительность или минимальный размер.
- Обычные `int`, `unsigned int` - для арифметики общего назначения, когда точный размер не важен.

Чтение бинарного файла с фиксированным форматом:
```
#include <cstdint>
#include <fstream>

struct Header {
    uint8_t magic[4];   // 4 байта сигнатуры
    uint16_t version;   // 2 байта
    uint32_t dataSize;  // 4 байта
};

int main() {
    Header h{};
    std::ifstream file("data.bin", std::ios::binary);
    file.read(reinterpret_cast<char*>(&h), sizeof(h));
}
```
Открывается файл `data.bin` для чтения в двоичном режиме (`std::ios::binary` отключает преобразования перевода строки и т.п.).

Метод `read` читает из файла сырые байты.

`reinterpret_cast<char*>` — говорит компилятору: считай это указателем на массив байт.

`sizeof(h)` — сколько байт прочитать (ровно размер структуры).

Итог: из файла читается блок байт и напрямую записывается в память переменной `h`. Так восстанавливаются значения полей `Header` - но только если структура строго соответствует бинарному формату (байт в байт).
Поэтому нужны типы `uint8_t`, `uint16_t`, `uint32_t`, чтобы гарантировать одинаковый размер и расположение на всех платформах.

# binary stream (`char*`)
Стандарт C и C++ определяет `char*` как указатель на байты памяти.

Ключевые причины:
1. Минимальная единица хранения — байт.
`char` (и `unsigned char`, `std::byte`) — единственные типы, которым разрешено напрямую указывать на “сырые” байты произвольных данных.

2. Правила строгой алиасинг-модели.
Доступ к объекту через указатель другого типа (например, `int*`, `float*`) может быть undefined behavior. Но через `char*` доступ всегда разрешён. Это стандартный способ читать или писать данные как последовательность байт.

3. Потоки (`istream`, `ostream`) работают с байтами.
`std::istream::read` принимает `char*`, потому что поток не знает, что вы читаете - числа, структуру или изображение. Он просто передаёт байты, а интерпретацию определяете вы.

```
int x;
std::cin.read(reinterpret_cast<char*>(&x), sizeof(x));
```
Поток заполняет память переменной `x` теми байтами, что пришли из файла. Если бы поток принимал `int*`, он не мог бы работать с произвольными данными.

# Сборщик мусора
В C++ нет сборщика мусора по умолчанию, потому что язык ориентирован на максимальный контроль и эффективность. А автоматизированное управление ресурсами решается другими механизмами (RAII, умные указатели). Существует сторонний Boehm GC, а также гибридные подходы (например, в Qt есть свой `QObject` с системой владения). Но стандарт языка не навязывает это, оставляя выбор программисту.

C++ изначально проектировался как расширение C. Главная идея: "добавить объектно-ориентированные возможности, не жертвуя контролем и эффективностью". C++ активно используется там, где важно точно знать:
когда освободится память, когда будет вызван деструктор (например, закрытие файла, мьютекса, сокета).

## Потенциальная реализация
1. Трассирующий сборщик (mark & sweep, как в Java). В памяти хранится куча объектов. Время от времени GC «обходит» все корни (стек, глобальные переменные, регистры) и помечает все достижимые объекты. Всё, что не помечено, освобождается.
Проблемы в C++:
   - Указатели — это просто адреса, и компилятор не знает, где именно в стеке или структуре лежит указатель, а где просто случайное число, похожее на адрес. Это делает трассировку ненадёжной.
   - Можно использовать консервативный GC (например, Boehm GC), который считает "любое значение, похожее на адрес" потенциальным указателем. Это работает, но не идеально.
2. Счётчик ссылок (reference counting, shared_ptr). Каждый объект хранит число владельцев. Когда счётчик обнуляется → память освобождается. Это уже встроено в стандарт (`std::shared_ptr`). Но не справляется с циклическими ссылками:
    ```
    struct Node {
    std::shared_ptr<Node> next;
    };
    auto a = std::make_shared<Node>();
    auto b = std::make_shared<Node>();
    a->next = b;
    b->next = a; // память никогда не освободится
    ```
    Для решения нужен std::weak_ptr или дополнительный сборщик циклов.
3. Региональная/аренная память (region-based / arena allocation). Объекты выделяются в "арене" (куче памяти). Когда арена больше не нужна, вся память освобождается сразу. Очень эффективно и предсказуемо (используется в игровых движках, компиляторах). Но это не «настоящий» GC — программист всё равно управляет временем освобождения.

## Пример простейшего "сборщика мусора" (на счётчиках ссылок)
```
#include <iostream>
#include <memory>

struct Node {
    std::string value;
    std::shared_ptr<Node> next;
    Node(std::string v) : value(std::move(v)) {}
};

int main() {
    auto a = std::make_shared<Node>("A");
    auto b = std::make_shared<Node>("B");
    
    a->next = b;
    b->next = a; // цикл, shared_ptr не освободит память!

    std::cout << "end\n";
}
```
Здесь будет утечка памяти. Если заменить одну из ссылок на `std::weak_ptr`, то цикл разорвётся и память освободится:
```
struct Node {
    std::string value;
    std::weak_ptr<Node> next; // слабая ссылка
    Node(std::string v) : value(std::move(v)) {}
};
```

# `std::byte`
Это тип из C++17, предназначенный для представления сырых байтов данных.

Это не числовой тип, в отличие от `char`, `unsigned char` или `uint8_t`. Он используется, когда нужно работать с памятью как с байтами, без арифметики. Над `std::byte` разрешены только побитовые операции (`|`, `&`, `^`, `~`, `<<`, `>>`), но не сложение или вычитание.

```
#include <cstddef>
#include <iostream>

int main() {
    std::byte b{0b00001111};
    b |= std::byte{0b11110000};
    std::cout << std::to_integer<int>(b); // выведет 255
}
```

Используется для:
- безопасной работы с необработанной памятью (`std::memcpy`, буферы);
- чёткой семантики: видно, что это именно байты, а не числа;
- API, где важно различать данные и их интерпретацию.
